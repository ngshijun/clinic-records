# Patient Tracking PWA — Design

**Date:** 2026-04-22
**Owner:** Shijun (Poliklinik Ng Plt)
**Status:** Spec approved — ready for implementation plan

## 1. Summary

A Progressive Web App for patients of a single clinic to keep a local, cloud-synced history of their vaccinations and blood tests, with push-notification reminders for follow-up doses or repeat tests. Records enter the PWA exclusively via QR codes generated by clinic staff at the point of care. The app has no appointment booking, no lab result values, no manual record entry, and no integrations with external health systems. Scope is deliberately narrow so the app stays focused on being a reliable, always-available record-keeper.

The system ships as one codebase with two faces: a patient-facing PWA at `/` and a staff-facing QR generator at `/staff` gated by a shared clinic password. The only server-side component is a single Supabase Edge Function that dispatches Web Push notifications on a schedule.

## 2. Goals

- Patients can view a chronological record of their vaccinations and blood tests, per profile, on any device they log into.
- Patients receive a push notification when a follow-up dose or repeat test is due.
- Clinic staff can generate a QR code at the end of a visit in under 30 seconds with minimal typing.
- A parent can track their own records plus their children's records on one device via profiles.
- The app works offline for reading; scanning and ingestion require connectivity.
- Zero operational overhead beyond Supabase — no Node server to maintain.

## 3. Non-goals (v1)

Explicit scope limits to prevent drift during implementation:

- No appointment booking or scheduling.
- No blood test result values (the PWA records *that* a test happened, not *what* it found).
- No manual record entry. QR is the sole ingestion path.
- No lot numbers, clinic names, or provider metadata in records.
- No staff-side record viewing, editing, or deletion.
- No PDF import, lab-report OCR, or photo attachments.
- No family-account sharing between Supabase users.
- No manual export/backup file — server is the source of truth.
- No multi-language support (English only).
- No staff-per-user accounts, no audit log.
- No cryptographic signing of QR payloads.
- No native Android or iOS app.
- No integration with MOH, national vaccine registry, or insurance systems.
- No SMS or email reminders — Web Push only.
- No undo or trash for deletes.

## 4. Architecture

### 4.1 Components

| Component | Runs on | Role |
|---|---|---|
| Patient PWA | Client device (browser or installed PWA) | UI, QR scanning, offline cache |
| Staff generator | Client device (clinic tablet) | Form → QR rendering |
| Supabase Auth | Supabase | Email+password, persistent sessions |
| Supabase Postgres | Supabase | Source of truth for all records |
| Edge Function: `send-reminders` | Supabase (Deno) | VAPID-signed Web Push dispatch |
| `pg_cron` + `pg_net` | Supabase | Schedules `send-reminders` every 15 min |
| Browser push service | FCM / APNs / Mozilla | Routes pushes to devices |

### 4.2 Stack

- **Frontend**: Vite + Vue 3 + Pinia + shadcn-vue + Tailwind CSS v4
- **Routing**: Vue Router (file-based via `unplugin-vue-router` optional)
- **PWA tooling**: `vite-plugin-pwa` (manifest + Workbox service worker)
- **QR scanning**: `html5-qrcode` or `@zxing/browser`
- **QR generation**: `qrcode` (npm package)
- **Auth + DB client**: `@supabase/supabase-js`
- **Edge Function**: Deno + `web-push` (via `npm:` specifier)
- **Hosting**: Any static host (Cloudflare Pages, Netlify, Vercel static). No Node runtime needed.

Rationale for Vite+Vue over Nuxt/Next/SvelteKit: the app is fully auth-gated (no SEO or SSR value), Supabase handles all server concerns (including VAPID dispatch via Edge Functions + `pg_cron`), and matching the `clavis` stack makes components and patterns directly reusable.

### 4.3 Data flow summary

1. Staff fills the generator form → QR rendered client-side (no server call).
2. Patient scans QR (in-app scanner or phone camera) → PWA parses URL fragment.
3. PWA inserts `records` row (and optional `reminders` row) via Supabase client; RLS scopes to the logged-in user.
4. `pg_cron` periodically wakes the Edge Function, which selects due reminders and sends VAPID-signed pushes.
5. Service worker receives push → shows OS notification → tap opens the PWA to the relevant record.

## 5. Data model

All tables enable Row-Level Security. Policy on every table is `user_id = auth.uid()`.

### 5.1 `profiles`

```sql
id            uuid primary key default gen_random_uuid()
user_id       uuid not null references auth.users(id) on delete cascade
name          text not null
date_of_birth date
notes         text
is_default    boolean default false
created_at    timestamptz default now()
updated_at    timestamptz default now()
```

### 5.2 `records`

```sql
id             uuid primary key default gen_random_uuid()
user_id        uuid not null references auth.users(id) on delete cascade
profile_id     uuid not null references profiles(id) on delete cascade
kind           text not null check (kind in ('vaccination','blood_test'))
name           text not null
performed_on   date not null
dose_number    int
total_doses    int
notes          text
qr_fingerprint text unique
created_at     timestamptz default now()
```

`qr_fingerprint` UNIQUE prevents exact-duplicate ingestion. `dose_number` and `total_doses` are NULL for blood tests.

### 5.3 `reminders`

```sql
id            uuid primary key default gen_random_uuid()
user_id       uuid not null references auth.users(id) on delete cascade
profile_id    uuid not null references profiles(id) on delete cascade
record_id     uuid references records(id) on delete cascade
kind          text not null check (kind in ('next_dose','followup_test','other'))
title         text not null
due_at        timestamptz not null
window_days   int default 7
sent_at       timestamptz
dismissed_at  timestamptz
completed_at  timestamptz
created_at    timestamptz default now()
```

`due_at` is computed client-side at scan time as `performed_on + nd days` at 09:00 in the scanning device's local timezone, then stored as `timestamptz` (UTC). `window_days` is a display-only hint used to phrase the UI as "due around <date> ± window_days"; it does not affect when the push fires (the push fires at `due_at` exactly, rounded up to the next 15-minute cron tick).

### 5.4 `push_subscriptions`

```sql
id            uuid primary key default gen_random_uuid()
user_id       uuid not null references auth.users(id) on delete cascade
endpoint      text not null unique
p256dh        text not null
auth          text not null
user_agent    text
last_seen_at  timestamptz default now()
created_at    timestamptz default now()
```

### 5.5 RLS policies

```sql
alter table profiles enable row level security;
alter table records enable row level security;
alter table reminders enable row level security;
alter table push_subscriptions enable row level security;

create policy "own rows" on profiles
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());
create policy "own rows" on records
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());
create policy "own rows" on reminders
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());
create policy "own rows" on push_subscriptions
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());
```

The Edge Function uses the `service_role` key, which bypasses RLS so it can scan reminders across all users.

## 6. QR payload format

### 6.1 URL shape

```
https://<app-domain>/ingest#v1.<base64url(json)>
```

- `#v1.` prefix declares the version; future versions use `v2.`, `v3.`, etc.
- Fragment (not query) so the payload never appears in server logs.
- `base64url` (no `+`, `/`, or `=`) keeps the URL clean.

### 6.2 JSON schema

```json
{
  "id": "01HYX3K5P2M4Q8W9R6T1Z7B0AE",
  "k":  "v",
  "n":  "Hepatitis B",
  "d":  "2026-04-22",
  "dn": 1,
  "td": 3,
  "nd": 30
}
```

| Key | Type | Meaning | Required |
|---|---|---|---|
| `id` | ULID string | Becomes `qr_fingerprint` | Yes |
| `k` | `"v"` \| `"b"` | `v` = vaccination, `b` = blood test | Yes |
| `n` | string | Display name | Yes |
| `d` | ISO date | Performed on | Yes |
| `dn` | int | Dose number | Vaccines only |
| `td` | int | Total doses in series | Vaccines only |
| `nd` | int | Next-due offset in days | Optional |

Typical payload ~180 bytes base64url. Fits comfortably in a QR that prints at a small size yet scans across a room.

### 6.3 Versioning

The `v1` prefix allows additive schema changes without breakage. On receipt, the parser dispatches to a version-specific handler; unknown versions show a friendly error.

## 7. User flows

### 7.1 Patient

**F1 — Sign up**
Email + password → create account → add first profile ("Me") → land on empty Home.

**F2 — Normal scan**
Home → Scan button → camera → scan QR → `/ingest#...` → profile picker (skipped if user has only one profile) → confirmation screen → `Add` → record + optional reminder inserted → redirect to record detail. If this is the user's first successful record, the notification-permission prompt appears immediately after the insert.

**F3 — View history**
Home shows Reminders (top) and History (below) for the active profile. Profile switcher in top bar.

**F4 — Edit / delete / move record**
Tap record → detail page → inline edit, `⋯` menu for **Move to profile** and **Delete**. Delete cascades to linked reminder.

**F5 — Receive reminder**
Push notification fires → tap opens PWA to the originating record. Actions: Dismiss, or scan the next dose when the time comes.

**F6 — New device**
Install PWA → sign in → Supabase client pulls everything. Push subscription is per-device; user re-enables notifications.

### 7.2 Staff

**S1 — First access**
`/staff` → shared-password gate. The expected hash is a PBKDF2-SHA-256 derivation (100k iterations, 16-byte salt) of the clinic password, stored in the frontend build's environment config (`VITE_STAFF_PASSWORD_HASH`, `VITE_STAFF_PASSWORD_SALT`). The page hashes the entered password with the same salt/iterations via WebCrypto and does a constant-time compare. On match, a `staff_unlocked` flag is written to `localStorage`; subsequent visits skip the gate until the flag is cleared by Logout.

**S2 — Generate a QR**
Form: kind, name (with autocomplete dictionary), date (default today), dose number/total, next-due days. Live QR preview on right with human-readable summary. `New QR` button clears form and generates a fresh ULID for the next patient.

**S3 — Regenerate for correction**
Editing any field auto-regenerates the QR. If the patient already scanned the prior QR, the similarity check on the next scan handles reconciliation.

### 7.3 Background

**P1 — Push dispatch cycle**
`pg_cron` every 15 min → calls `send-reminders` Edge Function → queries `reminders` where `due_at <= now() AND sent_at IS NULL` → sends VAPID-signed push per subscription → marks `sent_at = now()`. On `410 Gone`, deletes the dead `push_subscriptions` row.

**P2 — Offline**
Service worker caches app shell. IndexedDB (managed by the Supabase client cache or a thin adapter) serves records when offline. Scanning/ingestion require connectivity.

## 8. Correction model

**Principle:** patients own their records entirely; staff own QR authoring only. Staff has no mechanism to reach into patient data — the architecture doesn't carry a patient identifier in the QR.

### 8.1 Dedupe layers

1. **Exact dedupe**: `qr_fingerprint` UNIQUE — same QR scanned twice is silently rejected with toast "already in history."
2. **Similarity check**: before insert, look for records matching `(profile_id, kind, name, performed_on)` within 24 hours. If found, show confirmation: **Replace existing / Keep both / Cancel**.
3. **Replace** = a `replace_record(old_id uuid, new_record jsonb)` Postgres RPC function that performs `DELETE` then `INSERT` atomically in one transaction. The PWA calls this via `supabase.rpc(...)` rather than two client-side operations, to avoid a half-applied state if one call fails. No `supersedes_id` column.

### 8.2 Correction flows

| Scenario | Resolution |
|---|---|
| Staff typed wrong data, patient already scanned | Staff regenerates corrected QR → patient scans → similarity check → **Replace** |
| Staff noticed after patient left | Contact patient; regenerate at next visit or direct them to edit inline |
| Patient scanned into wrong profile | Record detail → `⋯` → Move to profile |
| Patient wants to delete a record | Record detail → `⋯` → Delete (hard delete, cascades to reminders) |
| Patient tries to delete their only profile | Blocked by client guard + a DB trigger that refuses when the profile is the user's last one — users must always have ≥1 profile to have a valid scan target |
| Missed scan | Staff regenerates same QR — stateless generation means no cost |
| Double scan of same QR | Silently caught by unique constraint |

## 9. Notifications

- **Permission prompt**: deferred until first record is ingested.
- **Subscription storage**: `push_subscriptions` table, one row per device. On logout, row deleted.
- **Dispatch**: `pg_cron` every 15 min → `send-reminders` Edge Function → `web-push` with VAPID keys from function secrets.
- **Content**: generic, lockscreen-safe (e.g., "Ahmad — Hep B Dose 2 due around today").
- **Click action**: opens `/records/:id` of the linked record.
- **iOS**: Web Push requires the PWA to be installed to home screen. Settings screen detects iOS Safari and shows an install prompt.
- **Cleanup**: `410 Gone` from a push endpoint → delete that subscription row.

## 10. Security

- **Patient auth**: Supabase email + password with persistent sessions (handled by `@supabase/supabase-js`).
- **Row-level security**: the whole security model. Every user-owned table enforces `user_id = auth.uid()`.
- **Staff gate**: shared password checked client-side against a hash. Weak-by-design; the real security boundary is the physical clinic tablet. Worst case if leaked: a stranger prints junk QRs that nobody scans.
- **Edge Function**: uses `service_role` key stored as a Supabase secret. Invoked only via `pg_net` from inside Postgres.
- **VAPID keys**: generated once, stored as Edge Function secrets.
- **QR payloads**: contain no PII — vaccine name, date, dose only. Harmless if dropped or photographed.
- **Records at rest**: plaintext in Supabase Postgres. No end-to-end encryption. Acceptable because (a) scope is a single owned clinic and (b) patient explicitly logs in to view, so the threat model reduces to "protect the Supabase project."

## 11. Deferred decisions

Captured here so they don't get lost — none block v1.

- **Soft delete / undo**: revisit if patients ask for a trash bin.
- **QR signing (HMAC)**: add if the app ever needs to prove a QR came from the clinic (e.g., for insurance).
- **Staff audit log**: add if multiple staff members need accountability.
- **Multi-language (i18n)**: add via `vue-i18n` following the `clavis-i18n-mandarin` pattern.
- **Family sharing**: explicit multi-user access to the same profiles — needs new access-control tables.
- **Manual record entry**: keep out of v1, but the addition is small if ever needed.
- **Result values for blood tests**: a new `record_results` table if/when requested.

## 12. Success criteria

The v1 is considered successful when:

- A staff member can generate and print a correct QR in under 30 seconds.
- A patient can scan and save a record in under 10 seconds on a typical phone.
- A patient reliably receives a push notification on the scheduled day for their follow-up dose, on both Android and installed iOS PWA.
- A patient who reinstalls the PWA on a new device sees all their history after logging in.
- A parent can track their own record plus two children's records without confusion (profile switcher works).
- The clinic has zero operational overhead beyond Supabase's dashboard.
